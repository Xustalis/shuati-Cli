# shuati-Cli v0.1.0 架构重构与升级计划
> 版本: v0.1.0 规划执行稿  
> 目标: 用全域智能引导式 TUI 替代纯文本 REPL，保留 CLI 兼容与脚本化能力。

## 1. 现状梳理与架构兼容性结论

### 1.1 项目架构与技术栈
- **分层结构**：CLI 层（`src/cmd`）→ 核心层（`src/core`）→ 基础设施（`src/infra`）→ 适配器（`src/adapters`）→ 工具层（`src/utils`）。
- **关键服务与业务目标**：题库管理、判题沙箱、AI 诊断、SM2 记忆系统、本地数据库与爬虫集成。
- **终端 UI 能力**：已引入 `ftxui` 并在启动引导与交互选择中使用，说明 TUI 与现有技术栈兼容。
- **交互入口**：当前无参数启动由 BootGuard + REPL 负责，命令式 CLI 由 CLI11 直接执行。

### 1.2 与升级计划的兼容性判断
- **兼容结论：可行，但需补齐配置与入口协议**  
  需新增 `config.ui_mode` 并与 `autostart_repl` 协同，保持 `legacy` 可默认回退。  
  需要在启动分支中新增 `TuiApp::run()`，并避免破坏原 CLI11 子命令入口与脚本使用方式。

## 2. 历史问题复盘与回归风险评估

### 2.1 已修复问题与可能回归点
- **数据库锁死**：已通过 WAL + 并发写保护优化；TUI 引入后台任务后需保证同样的写入隔离。
- **配置文件覆盖**：已改为 Load-Merge-Save；新增 `ui_mode` 时必须沿用该机制。
- **Windows 路径编码**：已统一 UTF-8 / W API；TUI 渲染与文件路径显示必须继续使用 UTF-8 转换工具。
- **REPL 启动策略**：已引入 `autostart_repl`；TUI 进入策略必须避免与之冲突并保持可开关。

### 2.2 用户痛点与计划覆盖度
- **痛点**：命令参数记忆、重复输入题目 ID、交互流程割裂。  
  **覆盖**：TUI 工作台与向导机制覆盖核心痛点，且保留旧版 CLI。

## 3. 功能清单、接口与依赖变更

### 3.1 功能列表（按里程碑）
- **M1：TUI 基础框架**：全屏框架、顶部状态栏、底部提示、主菜单。
- **M2：题目工作台闭环**：题目选择器、左右分栏、编辑/测试/AI/提交闭环。
- **M3：向导化入口**：新建题目、拉取题目、配置向导。
- **M4：统计复盘与视觉优化**：掌握度统计、配色可读性增强。

### 3.2 接口与配置变更
- **新增配置**：`ui_mode: tui | legacy`，默认为 `legacy` 以保证兼容。
- **启动策略**：
  - `argc > 1` → CLI11 原流程不变  
  - `argc == 1` → `BootGuard` → `ui_mode` 分流  
  - `ui_mode=tui` → TUI 主循环  
  - `ui_mode=legacy` → REPL  
- **服务层复用**：继续通过 `Services::load` 获取 `ProblemManager/Judge/AICoach/Database`。

### 3.3 依赖版本与兼容性
- 依赖由 vcpkg 锁定（包含 `ftxui`、`replxx`、`cpr`、`SQLiteCpp`）。
- 不引入新 UI 框架，避免额外依赖冲突。

## 4. 可行性矩阵（含量化依据）

| 模块 | 风险 | 量化依据 | 关键控制点 |
|---|---|---|---|
| M1 TUI 框架 | 低 | 现有 `ftxui` 已在 BootGuard 与部分交互使用 | 启动链路不破坏 CLI11 |
| M2 工作台闭环 | 中 | 需整合判题/AI/提交流程，涉及多 IO 与状态同步 | 任务队列与状态机隔离 |
| M3 向导 | 中 | 涉及爬虫与配置写入 | 严格复用现有命令逻辑 |
| M4 统计复盘 | 中 | 依赖数据库统计与渲染性能 | 统计缓存与分页展示 |
| 配置迁移 | 低 | `config.json` 已支持 Merge Save | 增量字段兼容 |
| 终端兼容性 | 中 | Windows/WSL/旧终端渲染差异 | 提供 `legacy` 回退 |

**性能基线与资源预算**
- **题目列表**：默认分页 100/页，单页渲染目标 < 16ms  
- **内存预算**：10k 题目元数据缓存 ≤ 40MB；TUI 常驻 ≤ 120MB  
- **并发上限**：交互任务单用户场景，UI 线程 + 最多 2 个后台任务（判题/AI）  
- **磁盘增长**：题目文本 + 测试用例数据保持现有路径，新增 TUI 缓存日志 ≤ 50MB/用户

## 5. 静态扫描与潜在瓶颈定位

- **数据库变更成本**：Schema 已较稳定，TUI 阶段避免新增表；统计应使用现有表 + 视图，避免迁移风险。
- **第三方库冲突**：`ftxui` 与 `replxx` 并存，需确保同一入口不同时初始化两者的交互循环。
- **API 往返次数**：AI 诊断流式输出长请求，TUI 必须后台线程执行并提供取消机制。
- **渲染阻塞点**：题目列表/测试结果长表渲染可能阻塞，需要增量渲染与缓存。
- **终端兼容性**：旧终端对宽字符/Emoji 支持不稳定，必须提供降级样式与无 Emoji 模式。

## 6. 优化策略与迁移方案

### 6.1 性能与稳定性优化
- **缓存策略**：题目列表与统计缓存 3s 刷新窗口，避免频繁 DB 查询  
- **异步拆分**：判题/AI 调用在后台线程执行，UI 主线程只处理渲染与事件  
- **渲染优化**：分页 + 搜索条件增量刷新，长文本折叠与按需展开

### 6.2 增量迁移脚本
- 仅新增 `config.ui_mode` 字段，使用原有 Merge Save 机制即可完成迁移，无需额外迁移脚本。

### 6.3 灰度发布与回滚
- **灰度路径**：`ui_mode=legacy` 为默认，新版本提示可试用 `ui_mode=tui`  
- **回滚策略**：任何 TUI 入口异常直接回退到 REPL，CLI11 命令保持不变

## 7. 可度量体验指标与 A/B 方案

### 7.1 指标
- 启动时长：冷启动 ≤ 1.5s，热启动 ≤ 0.8s  
- 交互响应：常规按键响应 ≤ 100ms  
- 稳定性：月度崩溃率 ≤ 0.1%  
- 快捷键一致性：主操作键位跨模块统一率 ≥ 95%  
- 配色对比度：文本对比度 ≥ 4.5:1

### 7.2 A/B 设计
- **分组**：A 为默认 legacy，B 为默认 tui  
- **观测指标**：首日完成题目数量、平均单题处理时长、中途放弃率  
- **评估周期**：两周，样本 ≥ 200 用户
